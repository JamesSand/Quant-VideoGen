# 三个视频生成模型的架构解析：它们是怎么生成"下一帧"的

> 基于本仓库集成代码的逐行侦察（file:line 均可核对）+ 各模型官方 model card / tech report。
> 配套文档：[REPORT.md](REPORT.md)（复现结论）、[EXPERIMENTS.md](EXPERIMENTS.md)（实验细节）。

## 0. 先建立共同底座：它们都不逐帧生成

三个模型共享同一套基础结构，差异只在"自回归的粒度和条件机制"：

```
文本/图像条件 ──┐
                ▼
  像素帧 ──VAE 编码──▶ latent 帧（时间 4× 压缩：1 latent ≈ 4 像素帧）
                ▼
        DiT（扩散 Transformer）在 latent 空间去噪
        每个 latent 帧被切成 ~880-1560 个 token 参与注意力
                ▼
  latent ──VAE 解码──▶ 像素帧（帧数 = 4L − 3）
```

**关键共性**：都是"**一次生成一块（chunk/段），块内联合去噪，块间自回归**"。没有任何一个是逐帧递归的。"下一块"的生成通过注意力读取**历史帧的 KV cache** 来保持连贯——这个 cache 就是 QVG 论文要量化的对象，也是长视频的显存瓶颈所在。

三者的本质差异一张表：

| | LongCat-Video | Self-Forcing | HY-WorldPlay |
|---|---|---|---|
| 自回归单元 | **1 段 = 20 新像素帧**（5 latent） | **1 块 = 3 latent**（12 像素帧） | **1 块 = 4 latent**（16 像素帧） |
| 每单元去噪成本 | **56 步** 完整扩散 + CFG | **4 步**（蒸馏） | **4 步**（蒸馏 Euler） |
| 历史条件 | 滑窗 73 像素帧（重叠式续写） | 全部历史（窗口内） | 记忆窗口内全部历史 + **动作信号** |
| KV cache 生命周期 | **每段推倒重建**（从像素重编码） | **持久流式**（跨块累积） | **持久流式**（跨块累积） |
| 生成速度 | ~6 分钟/段（H100） | ~16 FPS 实时流 | 准实时（4 步/块） |

## 1. LongCat-Video：段式续写（重叠窗口 + 段内双向扩散）

**范式**：把"长视频"化为"反复做视频续写"。模型本体是个标准的 13.6B 文生视频 DiT（默认一次生成 93 帧），长视频靠外层循环。

```
初始视频（93 帧，由一次普通 T2V 生成）
   │
   ▼  每段循环（run_long_t2v.py:208-313）：
┌──────────────────────────────────────────────────────┐
│ ① 取当前视频最后 73 帧 ──VAE──▶ 19 个条件 latent      │
│ ② 一次"cache 构建前向"：把条件 latent 过 DiT，        │
│    存下 48 层 × (K,V)（29,640 token/层，22.3GB bf16） │  ← QVG 在这里量化
│ ③ 从纯噪声起，56 步去噪 5 个新 latent（=20 新帧）：   │
│    每步、每层的注意力 = 新 token 自注意                │
│                + 读条件 cache（cross 到历史）          │
│    段内 5 个 latent 是【双向注意力联合生成】的，       │
│    没有先后次序 —— 这就是"一次 20 帧"                  │
│ ④ VAE 解码新 latent → 20 帧接到视频尾部               │
└──────────────────────────────────────────────────────┘
   │ 滑动窗口，回到 ①（cache 完全丢弃、下段重建）
   ▼
```

**架构特点与后果**：
- 段内是**完整的高质量扩散**（56 步 + CFG 双 batch），所以单段质量最高、也最慢（~6 min/段）。
- cache 每段从像素重建 → 显存恒定（22.3GB），**长度无上限**；代价是历史信息经过"latent→像素→latent"往返，只剩 73 帧窗口。
- 逐帧保真度呈段内梯度：紧贴条件窗的帧被"钉"得最牢（我们实测首帧 PSNR 29+），离窗口越远自由度越大（第 20 帧 ~15）。
- 官方 demo 用 13 帧重叠；本仓库集成改成 73 帧（更强条件、cache 大 5 倍——更能凸显量化收益）。

代码入口：`experiments/LongCat/run_long_t2v.py`（外层循环）、`longcat_video/pipeline_longcat_video.py:1160-1195`（prepare→cache→quantize→denoise）、`longcat_video/modules/attention.py:271`（读 cache 的注意力，pre-RoPE 存 K）。

## 2. Self-Forcing：因果流式（3-latent 小块 + 4 步蒸馏 + rolling cache）

**范式**：把 Wan2.1-T2V-1.3B 蒸馏成**因果化的流式生成器**——训练时就按"块状自回归 + KV cache"的推理方式训练（消除 train-test gap，这是论文标题的含义），实现 ~16 FPS 实时生成。

```
每块循环（pipeline/causal_inference.py）：
┌────────────────────────────────────────────────────────┐
│ ① 取 3 个 latent 帧的纯噪声（4,680 token）              │
│ ② 4 步去噪 [1000→750→500→250]：                        │
│    每步：新 token 自注意 + 【因果】读全部历史 KV cache  │
│    （K 以 pre-RoPE 形式存，读时对全长重新加 RoPE）      │
│ ③ 第 5 次前向：把去噪完成的干净块以 t=0 重新编码，      │
│    其 KV 追加进 cache（供后续块读取）                   │
│ ④ 攒够若干块后 VAE 解码输出 12 帧                       │
└────────────────────────────────────────────────────────┘
   │ cache 持续增长（或滑窗淘汰），回到 ①
   ▼
```

**架构特点与后果**：
- 每块只有 4+1 次前向 → 唯一能实时流式的模型；代价是单步质量靠蒸馏（DMD）保证。
- **官方训练视野只有 21 latent（5 秒）**，原版代码硬性封顶（`max_attention_size=32760`）。更长 = 靠 rolling KV cache 外推（"原则上无限"，作者自己标注超长会退化）。本仓库集成把窗口开到全长（`local_attn_size = 总长`），所以 cache 随长度线性膨胀——1.3B 的小模型却有三者中**最重的 cache**（256MB/latent，180 latent 就 46GB），因为它不做任何历史压缩。
- 块小、步少 → 误差按块高频累积，漂移最快（我们的 PSNR 曲线在 SF 上衰减最陡）。

代码入口：`experiments/Self-Forcing/pipeline/causal_inference.py`（块循环 + 每 8 块量化一次 cache）、`wan/modules/causal_model.py:272-313`（`attn_kv_cache_prerope`：pre-RoPE 存取 + 全长 RoPE 重应用——我们极限实验中 OOM 的现场）。

## 3. HY-WorldPlay：动作条件世界模型（4-latent 块 + 记忆窗 + 相对位置）

**范式**：不是"文生视频"而是"**可交互世界模拟**"：给一张初始图 + 一串动作指令（前进/后退/转视角……），模型像游戏引擎一样逐块渲染世界对动作的响应。

```
初始图 ──VAE──▶ latent₀ 入 cache；全部 241 latent 的噪声在第 0 块一次性预采样
   │
   ▼  每块循环（pipeline_wan_w_mem_relative_rope.py）：
┌────────────────────────────────────────────────────────┐
│ ① 取本块动作信号（如 "w-8" = 前进），注入为条件嵌入     │
│ ② 从记忆窗口取历史 latent 的 KV cache，                 │
│    用【相对 RoPE】重排位置（历史永远"贴着"当前块，      │
│    避免绝对位置外推失真）                               │
│ ③ 4 步 Euler 去噪本块 4 个 latent（=16 像素帧）         │
│    每 2 块把老历史量化一次（QUANT_FACTOR=2）            │  ← QVG 挂点
│ ④ 新块 KV 入 cache；VAE 解码输出 16 帧                  │
└────────────────────────────────────────────────────────┘
   │ 回到 ①（下一个动作）
   ▼
```

**架构特点与后果**：
- **动作是第一类条件**：同一历史 + 不同动作 → 不同的下一块。这让它成为三者中唯一"可交互"的（官方在线 demo 就是实时游玩）。
- 官方版的长期一致性靠"**重构式上下文记忆**"：固定只看 4 chunk（3 个最近 + 1 个按几何相关性从远处检索的），配合位置重排，计算量不随长度增长。**本仓库集成简化成了大连续记忆窗**（48-244 latent 全保留）——所以显存随窗口线性涨，这是我们测到 bf16 317 帧显存墙的直接原因（官方架构本不该有这堵墙）。
- 注意标注差异：官方 8B 版基于 HunyuanVideo-1.5 骨干；本仓库集成加载的是 **Wan2.2-TI2V-5B** 组件（QVG paper 写"HY-WorldPlay-8B"，与实际集成的 5B Wan 变体存疑）。

代码入口：`experiments/HY-WorldPlay/wan/generate.py`（动作解析 + 块循环）、`wan/inference/pipeline_wan_w_mem_relative_rope.py:1058-1200`（记忆选择 + 相对 RoPE + 量化调度）。

## 4. 一图总结：三种"下一帧"哲学

```
LongCat     ████████████████████░░░░░  「重叠续写」：每次重看最近 73 帧，
            └─73 帧条件─┘└20 新帧┘      重新铺开一次完整扩散。质量优先。

SelfForcing ▶▶▶▶▶▶▶▶▶▶▶▶▶▶▶▶▶▶▶▶▶▶    「因果流水」：3-latent 小块贴着历史
            每块 4 步，cache 只增不删     cache 快速推进。速度优先。

HY-WorldPlay ⟳ 动作→块 ⟳ 动作→块 ⟳     「世界模拟」：动作驱动，记忆窗 +
            记忆窗口 + 相对位置          位置重排维持空间一致性。交互优先。
```

**对量化研究的含义**（为什么这三个恰好是 QVG 的测试床）：
- LongCat 代表"**大 cache、一次性量化**"场景（29,640 token 整窗量化，K=256 质心统计充分）；
- Self-Forcing 代表"**流式增量量化**"场景（每 8 块一次，新块永远 bf16、老块 INT2 的混合读取）；
- HY-WorldPlay 代表"**持久记忆量化**"场景（量化误差会通过记忆窗被反复读取、跨块传播）。
- 三者的共同软肋也一致：量化误差经自回归反馈被混沌放大（见 REPORT §二），且发布实现的读取路径按全长 bf16 反量化，使 INT2 的长度解锁停在 ~3× 而非 7×（见 REPORT §三-bis）。
